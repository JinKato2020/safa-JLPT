# 合格リング（合格率）エビデンスモデル再設計 — 設計書

**日付**: 2026-07-16
**対象**: 合格リング（ホームの「合格率％」大リング）の"％の作り方"の再設計
**関連メモ**: pass-ring-evidence-model / practice-session-question-counts / dynamic-choices-distractor-pool / ladder-facet-mastery-analysis-first

## 1. ゴール
合格リングの中央値（合格率％）を、**3つの学習信号を統合**して算出する：
1. **練習**（試験タブの練習）＝日々の主信号。ただし**反復による水増しを割り引く**。
2. **模試**（本番形式）＝**常に初見**の最も信頼できる信号。**重みを最大**にする。
3. **単語タブ**の学習＝**面別マスタリー（語×面）**経由で対応大問を底上げ。

現状は「模試の扱いがIDリネームで壊れ」「単語タブが中央％に効かない」「模試/練習の重み差が中央％に無い」。これを恒久的に作り直す。

## 2. 現状（コード実測・2026-07-16）
- **中央％** = `src/store/selectors.ts` `ladderPassPct`。各大問プール（`daimonUnitIds(level,daimon)`）の `state.items[id].p` を単純平均→ladderモンテカルロ（`ladder/passRate.ts`）。**記録の由来（練習/模試）を区別せず平等**。
- **信頼幅±** = `src/engine/engine.ts` `computeReadiness`。`band = 11·exp(−evidenceTotal/80)+未測定加算`。`evidenceTotal` は**全 `state.items` の evidence 合算＝単語タブ（`#produce`/`#gbuild`/`#gmeaning`）も含む**→単語タブ学習は±を狭めるが中央％は不変。engine内部に evidence 重み `mock:5`／練習軽 は**存在するが±側のみ**。
- **記録経路**：`store.tsx` の `QUIZ_ANSWER`（練習）は常に `state.items` へ記録。`MOCK_ANSWER`（模試）は `startsWith('kb-')` の台帳問題を「学習日のみ・evidence非記録」で除外。
- **不具合**：用法IDを `usg-` にした結果、`startsWith('kb-')` を外れ、(1)模試で用法がevidence記録され兄弟(組み立て/文章=kb-)と不整合、(2)`skillWeight` が級をID接頭辞で誤読しN5固定。N4(usg4-)にも波及。
- **単語タブ**は別キー空間（`#produce`等）＝中央％に無影響（±のみ間接影響）。

## 3. 新設計

### 3.1 面別マスタリー（語×面）を土台にする
- 各「語×面（大問）」に習得度 `p`（0–1）と `evidence`（累積重み）を持つ（既存 `ladder/mastery.ts` の考え方を踏襲）。
- 大問の到達度 = その大問に属する語×面の**カバー率×習得**（未着手=0で薄まる＝「全部覚える」が目標。既存 `knowledgeDaimonPct` を継承）。
- **単語タブの学習**：語 v を単語タブで習得→その語が関わる大問面（漢字読み/文脈規定/言い換え/文法 等）の `p` を底上げ。実装＝単語タブの `#produce/#gbuild/#gmeaning` 更新を、対応する試験大問の語×面マスタリーに**写像**（対応表は §3.5）。

### 3.2 由来別の重み（模試 > 練習）
各解答は**由来重み w** で evidence を積む：
- **模試 w=5**（最も信頼・初見保証）。※engine の既存 `mock:5` と整合。
- **練習 w=1**（基準）。
- **単語タブ w=1**（面写像経由・練習と同等）。（ユーザー確定2026-07-16）

習得度 `p` はロジスティック更新、`evidence` は `min(累積, EVIDENCE_CAP)` で飽和（既存踏襲）。**中央％の `ladderPassPct` は、単純平均でなく evidence 重み付き**で各項目の `p` を集約するよう変更（模試を受けた項目が中央％を強く動かす）。

### 3.3 練習の反復割引
同一項目を練習で繰り返すほど**正解率が上がる（実力過大評価）**ため、練習の寄与を逓減：
- **n回目の練習の寄与重み = 1 / (1 + k·(n−1))**（推奨 k=0.5：1回目1.0→2回目0.67→3回目0.5…）。
- または「項目あたり練習evidenceに上限（例 3.0）」を設け、それ以上は学習日のみ。
- **模試は割引しない**（毎回初見のため）。

### 3.4 模試＝完全独立プール・常に初見（生成は後続フェーズ）
- 模試は**練習に絶対出ない専用問題**（ID名前空間で分離。例 `mk-...`）。年6回想定・半年で忘却→再利用可・6回分備蓄で約1年無重複（詳細は問題側spec B / 後続）。
- 模試の各解答は §3.2 の w=5 で中央％に反映（**初見が保証されるので正当**）。
- **本specでは記録経路とプール分離のインターフェースのみ定義**。模試問題の生成は保留（練習の公式準拠生成体制が整ってから一括）。

### 3.5 記録ロジックの書き直し（usg-不具合の恒久修正）
`startsWith('kb-')` を**「台帳項目か（bank所属）」判定に一般化**：
- 共通ヘルパ `isBankUnitId(id)`（`daimon.ts`：`BANK_LEVEL.has(id)`）を新設。
- `store.tsx MOCK_ANSWER`：**「その項目が初見か（`!state.items[id]`）」で分岐**。初見→由来重み w=5 で記録（中央％へ）。既出（万一の再利用）→学習日のみ。※台帳/非台帳を問わず統一。
- `selectors.ts skillWeight`：級は**`bankLevelOf(id)`（=levelタグ）を優先**、無ければ接頭辞（n3/n4/n5）。→ `usg-`/`mk-` も正しい級。
- これで N3/N4 の用法（usg-）と将来の模試（mk-）が正しく級付け・記録される。

### 3.6 ±（信頼幅）の扱い
- 現状 `evidenceTotal` は単語タブも合算＝単語タブ学習で±が狭まる。**方針：由来重み付き evidence を用い、模試で強く収束・練習/単語タブは弱め**に（過信防止）。
- 単語タブの±寄与は残す（学習量の反映）が、係数を下げる（推奨：単語タブ evidence は×0.3 で band 計算へ）。レビューで調整。

### 3.7 コールドスタート（模試未受験）
- 模試ゼロでも**練習＋単語タブで中央％が動く**（§3.1–3.3）。模試を受けると w=5 で一気に精度が上がり±が縮む。
- 未測定大問は推測下限 `0.25`（既存）。
