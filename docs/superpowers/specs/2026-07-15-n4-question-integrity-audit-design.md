# N4 問題 整合性監査・本番同等化 設計書

> **これは何**: N4の3大問(用法・並べ替え・文章の文法)を「本番同等の弁別力」まで引き上げる監査＋自動修復パイプラインの設計。10言語展開の前に**日本語を確定(ロック)**し、翻訳の手戻りをゼロにするのが目的。
> **作成**: 2026-07-15 / スコープ確定: N4のみ(N3は模試情報待ちで保留)・名指しの3大問のみ・検出→自動修復。

---

## 1. 目的 / 非目的

**目的**
- N4の3大問の**一意性破綻**(正解が1つに定まらない)と**ダミー近接化不足**(当てずっぽうで消せる)を全数監査し、**自動修復**して本番同等品質にする。
- 構造的赤旗(正解位置偏り・選択肢重複・ルビ・長さtell・個人名混入)を機械検査で全数是正。
- 日本語を確定させ、以降の10言語翻訳が**再翻訳不要**になる状態を作る。

**非目的(このスコープ外)**
- N3・N5の同大問の監査(N3=保留、N5=次段で同じ型を適用)。
- 他の4択大問(文脈規定・言い換え・文法形式判断)のダミー近接化(今回は名指し3大問のみ)。
- **翻訳(解説等の9言語化)そのもの**。翻訳は別フェーズ・有料で、円換算見積り→承認を別途取る。本設計は「翻訳前の日本語ロック」まで。
- アプリUI/ローダの変更(既存 loader / rebuild をそのまま使う)。

### C1 位置偏りは無害(実測)
3大問すべて表示時に `shuffleChoices()` を通す(bank=daimon.ts:155、passage=PassageSetPlayer.tsx:27)。データ上「正解が先頭に偏る」(用法100%・文章の文法81.5%が位置0)は**実行時にシャッフルで打ち消される→C1は欠陥ではない**。C機械監査の実質指摘は **C5 個人名10件(用法7・並べ替え3=すべて『田中』)** のみ。他(C2重複/C3ルビ形式/C6整合)は全0。

- **LLM判定単位 = 805**(用法630＋並べ替え135＋パッセージ40)。個別問数=965。
- **翻訳同期の注意**: 用法・並べ替えは未訳なので修復しても訳の陳腐化なし。**文章の文法は既にネパール訳あり(パッセージ本文 `i18n.ne.body` のみ。空所選択肢200は全て未訳)**。→ **選択肢の差し替え(ダミー近接化)は訳に無影響**。**本文(body)を修復した時だけ**当該パッセージ訳が陳腐化 → 「再訳要」フラグ。修復の大半はダミー差し替えなので訳影響は最小。

### データ・スキーマ(実測)
- 用法/並べ替え(flat): `{id, stem, question, answer, choices[4], i18n:{}}`。`answer` は `choices` のいずれかと一致(order は★位置のタイル文字列)。
- 文章の文法(nested): `{id, level, kind, passages:[{body}], questions:[{id, blankNo, choices[4], answerIndex, pointId, i18n}], i18n}`。1パッセージ=本文＋5空所。
- スキーマ定数: `app/tools/content/schema.ts`(`DAIMON_SPEC` / `LANGS`)。検証: `app/tools/content/validate.ts`。manifest(OTA sha)再生成: `app/tools/content/rebuild.ts`。

## 3. 欠陥タクソノミー(監査で捕まえる対象)

- **A. 一意性破綻**(最重要・LLM判定):正解が1つに定まらない。大問別基準は §5。
- **B. ダミー近接化不足**(LLM判定):ダミーが表層で消せる。全て実在するN4語/N4文型で、**狙いの知識でのみ**消えるべき。表層の手がかり(長さ・レジスタ・明白なナンセンス)で消せてはいけない。
- **C. 構造的赤旗**(機械検査=Python・全数・¥0):
  - C1 正解位置の偏り(ファイル単位で `answer`/`answerIndex` 分布がほぼ一様か)。
  - C2 選択肢重複(同一item内に等価な選択肢)。
  - C3 ルビ完全性(級相当以上の漢字にルビ `漢字（かな）` があるか＝形式検査)。
  - C4 長さtell(正解だけが系統的に最長/最短でないか)。
  - C5 個人名混入(役割ベース違反。姓名リスト＋敬称パターンで検出)。[[content-borderless-no-names]]
  - C6 答えの整合(order: `answer`∈`choices`。passage: `answerIndex`∈0..3、`choices` 長4)。

## 4. パイプライン(4段・方式①)

```
[1] C機械監査 (tools/audit_questions.py・Python・全805単位・¥0)
     → structural_report.json: {id, checks:{C1..C6}, autofixable, flagsForLLM}
     → 自動修正できるC(位置リバランス/ルビ付与/重複解消の候補)は差分案として出力
        ↓
[2] A/B LLM監査＋自動修復 (in-session サブエージェント ~34体・大バッチ・自己検証内包・effort=high)
     入力: 各バッチのitem配列(argsで直接渡す。read:エージェント禁止[[workflow-subagent-efficiency]])
     各itemで: 一意性判定(§5) + ダミー近接性判定 → 欠陥なら修復案を同一応答で生成し
               その場で妥当性を自己検証(order=順列列挙 / passage=全選択肢の文脈適合 / usage=4文の正誤)
     出力(構造化): §6 の AuditVerdict[]
        ↓
[3] 修復問のみ 独立再検証 (in-session 少数・effort=high・敵対的「一意性を崩せるか」)
     修復されたitemだけを別エージェントが再判定。崩せたら[2]へ差し戻し(最大2周)。
        ↓
[4] 反映＆再ビルド
     承認済み修復をJSONへ適用 → node --import tsx app/tools/content/rebuild.ts で manifest(OTA sha)更新
     → validate.ts 通過確認 → 差分サマリ＋各大問サンプルをユーザーへ提示(実機確認用)
     → 文章の文法で**本文(body)を変えた**パッセージのみ「ネパール再訳要リスト」を別出力
```

### 効率規定(遵守)
- **read:エージェントを作らない**。データは `args` で直接渡す。[[workflow-subagent-efficiency]]
- **1バッチ=1エージェントの細粒度分割をしない**。**~30体の大きめエージェント**に束ねる(805単位 ÷ ~28単位/体 ≈ 29体)。
- **無条件の独立verify段を全数に付けない**。[2]で自己検証を基本とし、[3]は**修復した問だけ**(欠陥率に比例したコスト)。
- money **¥0(定額内・in-session生成)**。消費するのはセッションquotaのみ → 見積り約0.5M tokens(quota上限を焼いた反面例の約1/8)。

### バッチ設計
- 大問ごとに分ける(判定基準が違うため)。用法630→~23体(28/体)、並べ替え135→~5体、文章の文法40パッセージ→~6体(7パッセージ/体、本文込みで重いため小さめ)。計~34体。
- **10%ごとにバックアップ**: 各エージェント完了ごとに部分結果を `scratchpad/audit/partial_*.json` へ追記保存。途中停止でも救済可能に(CLAUDE.md規定)。

## 5. 大問別 一意性判定基準(監査プロンプトの核)

### 5.1 並べ替え(order)
- フレーム(固定テキスト＋★位置)と4タイルに対し、**文法的かつ意味的に成立する語順が厳密に1通り**であること。
- 判定: LLMが妥当な語順を列挙。2通り以上が成立 → 欠陥A。
- ★位置のタイルが**狙いの文法点**(例: のに/ように/たがって等)になっていること。
- 修復: 1タイルの助詞/語を差し替えて代替解を消す。★位置の狙いは保つ。再列挙で1通りを確認。
- 併せてC(★位置に来るタイルの `answer` 整合、ルビ)を確認。

### 5.2 文章の文法(passage_grammar)
- **本文全体の時制・人称・論理・結束性**の下で、各空所に適合する選択肢が**1つだけ**であること。
- 判定: LLMが本文を読み、各空所で4選択肢の文脈適合を個別評価。2つ以上適合 → 欠陥A。
- 修復: 過剰に成立するダミーを、**同カテゴリ(品詞・機能)だが文脈上明確に誤り**の語へ差し替え。`answerIndex` は保持(差し替えるのはダミー側)。**修復は原則ダミー(選択肢)側に閉じる**=本文を極力変えない。
- **訳同期**: 空所選択肢は未訳のため**選択肢差し替えは訳に無影響**。本文(body)を変えた時**だけ** `ne.body` を「再訳要」に。

### 5.3 用法(usage)
- 正解文だけが**正用かつ自然**、他3文は**明確な誤用**(単に不自然ではなく、用法として誤り)であること。
- 判定: LLMが4文を個別評価。欠陥は (a)2文以上が許容 (b)正解文が不自然/誤り (c)ダミーが実は正用、のいずれか。
- 修復: 該当文を書き直す。**ダミーは近接類義語の置換型を最優先**(同じ意味フィールドの近接類義語がそれぞれ最適になる文脈＝対象語だけ不自然。例: 届ける→のせる/渡す/送る)。共起・自他・格・レジスタのズレも可。単なる対義語・無関係語・使い回しは不可。正解文が問題なら自然な正用へ。
- 語(stem)自体は変えない(語の入替は別語との重複を生むため)。

### 共通(B・全大問)
- ダミーは全て実在するN4語/N4文型。狙いの知識でのみ消える。
- 個人名は使わず役割ベース(先生/学生/店員/客等)。[[content-borderless-no-names]] [[unique-answer-question-design]]

## 6. LLM監査の出力契約(構造化・スキーマ)

各エージェントは判定結果を次の配列で返す(パースせず検証層で受ける):

```jsonc
// AuditVerdict
{
  "id": "kb-000654",            // usage/order は item.id、passage は question.id(pg-...-qN)
  "daimon": "usage",
  "ok": false,                   // true=欠陥なし(修復なし)
  "defects": ["A_uniqueness"],   // ["A_uniqueness","B_distractor","C_*"] の部分集合
  "reason": "選択肢2と正解が両方この語の正用になっている",
  "repair": {                    // ok=true の時は null
    "stem": "…",                 // 変更したフィールドのみ(未変更は省略可)
    "answer": "…",
    "choices": ["…","…","…","…"],
    "answerIndex": 0             // passage のみ
  },
  "selfCheck": "修復後: 正解のみ正用、他3文は誤用(格/共起)。順列列挙で1通りを確認",
  "needsRetranslate": false      // passage で body/選択肢を変えた時 true
}
```

- 検証層(Python)が受け取り: `repair` の `choices` 長=4・`answer`∈`choices`(order)・`answerIndex`∈0..3(passage)・ルビ形式・個人名なし・重複なし を**機械再チェック**。落ちたら [3] へ回す。
- `ok=true` は無変更。`ok=false` かつ `repair` 有りのみ差分適用。

## 9. 受け入れ基準(このスコープの完了条件)

1. N4の3大問 全805単位がLLM判定を1回以上通過(サンプリングなし。打ち切り時は残数を申告)。
2. 検出された欠陥が自動修復され、修復問は独立再検証を通過(崩せない)。
3. 機械検査C1〜C6が全数グリーン(位置分布ほぼ一様・重複0・ルビ形式OK・個人名0・答え整合OK)。
4. `validate.ts` 通過、`rebuild.ts` で manifest 更新済み(OTA配信可能)。
5. 差分サマリ＋各大問サンプル(修復前後)をユーザーへ提示済み。
6. 文章の文法の「ネパール再訳要リスト」を出力済み(翻訳フェーズへの申し送り)。

## 10. リスク & 対策
- **修復が新欠陥を生む** → [3]独立再検証＋検証層の機械再チェックで二重に防ぐ。
- **quota逼迫** → 大バッチ・~34体・自己検証内包で約0.5Mに抑制。10%ごとバックアップで途中停止も救済。
- **訳の陳腐化** → passage修復に再訳要フラグ。用法/並べ替えは未訳のため影響なし。
- **過剰修復(良問まで書き換え)** → `ok=true` は無変更を厳守。修復は欠陥itemのみ。

## 11. コスト
- 監査・修復・再検証は **in-session生成=money ¥0(定額内)**。消費はセッションquotaのみ(~0.5M)。
- 翻訳フェーズ(別・有料)は本スコープ外。着手時に円換算見積り→承認。
